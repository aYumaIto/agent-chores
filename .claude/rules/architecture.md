# アーキテクチャ ルール

## モジュール設計の原則

- 各モジュール（クラス、関数、コンポーネント）は **単一責務** を持つ。複数の責務を混在させない
- 公開 API を提供する層と、内部ロジックを担う層を分離する
- テスト容易性のために抽象（インターフェース、型、Props）で依存を定義し、具象への直接依存を避ける

## 依存ルール

- 依存の流れは **一方向のみ** にする
- **循環依存は禁止**
- 逆方向の連携が必要な場合は、直接参照ではなくインターフェース、コールバック、イベント、または状態管理を使う
- 具象ではなく抽象に依存する（依存性逆転の原則）
- 依存オブジェクトはコンストラクタインジェクション、Props、または引数で受け取る

## 変更ガイドライン

- 外部向け公開 API の変更は、エントリポイント（クラス、エクスポート関数、ページコンポーネントなど）のみで行う
- 内部ロジックの変更は、該当する責務のモジュールに閉じる
- 新しい責務が必要な場合は、既存モジュールを肥大化させず新規モジュールを作成する

## 非同期処理・スレッドセーフティ

- バックグラウンド処理や非同期処理の結果は、適切なコンテキスト（UI スレッド、メインスレッド、レンダリングサイクルなど）にディスパッチしてから使用する
- スレッド切り替えや状態更新のディスパッチ処理は削除しない
