# Issue 作成

## 利用する場面
- GitHub Issue を新規作成するとき
- 既存 Issue の本文を編集するとき

## 必須ワークフロー

Issue の作成または編集では、必ず次の順序で進めます。

1. ファイル編集機能でドラフトファイルを作成または更新する（`gh` は先に実行しない）
2. ドラフト内容をユーザーに提示し、明示的な承認を得る
3. 承認後にのみコマンドを実行する。作成は `gh issue create`、編集は `gh api` の REST 呼び出しを優先する

承認ステップは絶対に省略しないでください。

## 内容正確性ルール

- Issue には問題点とあるべき姿、または目的のみを記載する。実装の詳細には立ち入らない
- 要件、受け入れ条件、スコープは、ユーザーが明示した内容のみを記載する
- 要求されていない機能、受け入れ条件、実装詳細を勝手に追加しない
- 重要事項が不足していると判断した場合は、追加前にユーザーへ確認する

## 手順

1. Issue のタイトルと内容を決める
2. `personal_workspace/` 配下に Issue ドラフトファイルを作成または更新する
3. ドラフトの本文を記述する
   <!-- TODO: プロジェクトの記述言語に合わせて編集してください -->
   <!-- 例: 英日バイリンガルの場合、各セクションを <English> --- <日本語> の構成にする など -->
4. ドラフトをユーザーへ提示し、明示承認を得る
5. 作成時は `gh issue create` を実行する
   ```bash
   gh issue create --title "<issue title>" --body-file <draft-file>
   ```
6. 編集時は `gh api` を使用する
   ```bash
   gh api repos/{owner}/{repo}/issues/{number} -X PATCH --raw-field body="$(cat <draft-file>)"
   ```

## 出力ルール

ドラフトファイルを作成または編集したら、ユーザーがすぐ開けるよう、チャット末尾に必ず Markdown リンクでファイルパスを表示します。
