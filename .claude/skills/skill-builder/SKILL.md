# スキル作成

## 利用する場面
- ユーザーから新しいスキルの追加を依頼されたとき
- 以下の **スキル化シグナル** を検出したとき（自動提案する）

## スキル化シグナル

次のパターンを検出した場合、スキル化をユーザーに提案すること。

| シグナル | 例 |
|---|---|
| 同じ手順を 2 回以上実行した | デプロイ手順、DB マイグレーション |
| 複数ステップを毎回同じ順序で実行する | lint → test → build → push |
| プロジェクト固有のコマンドやオプションを毎回調べ直す | 特殊なビルドフラグ、環境変数の組み合わせ |
| ユーザーが「いつもの手順で」と指示する | 定型ワークフローの暗黙的な参照 |

### 提案テンプレート

シグナルを検出したら、次の形式で提案する。

```
このワークフローをスキルとして登録しませんか？

- スキル名: `<skill-name>`
- 目的: <ワークフローの概要>
- 含まれる手順: <ステップの要約>
```

ユーザーが承認したら、下記の手順でスキルを作成する。

## スキルの構成

各スキルは `.claude/skills/<skill-name>/SKILL.md` に配置する。

```
.claude/skills/<skill-name>/
└── SKILL.md          # スキル定義（必須）
└── *.sh / *.py ...   # 補助スクリプト（任意）
```

## SKILL.md のフォーマット

```markdown
---
name: <skill-name>
description: "<スキルの概要。エージェントが利用場面を判断できるよう、目的と用途を簡潔に記述する>"
argument-hint: "<引数のヒント（任意）>"
---
# <スキルタイトル>

## 利用する場面
- このスキルを使うべき具体的な場面を箇条書きで記載する

## 手順

1. ステップを順番に記載する
2. コマンド例はコードブロックで記載する
3. プロジェクト固有の設定が必要な箇所は `<!-- TODO: ... -->` コメントで示す

## 注意

- 制約事項や注意点を記載する

## 出力ルール

ファイルを作成または編集したら、ユーザーがすぐ開けるよう、チャット末尾に必ず Markdown リンクでファイルパスを表示します。
```

## 手順

1. スキル名と目的を決定する（提案時は検出した内容から自動決定）
2. `skill-name` は kebab-case にする
3. 既存スキル（`.claude/skills/`）を確認し、名前の重複がないことを検証する
4. `.claude/skills/<skill-name>/SKILL.md` を上記フォーマットに従って作成する
5. 以下の品質チェックを行う：
   - `description` がエージェントの利用判断に十分な情報を持つか
   - 「利用する場面」が具体的か（曖昧だとエージェントが誤判断する）
   - 手順が再現可能な粒度で記載されているか
   - プロジェクト固有の箇所に `<!-- TODO: ... -->` コメントがあるか
   - ユーザー承認が必要なステップが明記されているか
6. 補助スクリプトが必要な場合は同ディレクトリに配置する

## 注意

- スキル化の提案はユーザーの承認を得てから作成する。勝手に作成しない
- 一度きりの手順はスキル化しない。繰り返し利用される見込みがあるものだけを対象とする
- 既存スキルと責務が重複する場合は、既存スキルの拡張を検討する

## 出力ルール

ファイルを作成または編集したら、ユーザーがすぐ開けるよう、チャット末尾に必ず Markdown リンクでファイルパスを表示します。
