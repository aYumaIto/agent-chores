# PR 作成

## 利用する場面
- 作業ブランチの変更を PR として提出するとき
- PR の本文を作成または編集するとき

## 必須ワークフロー

PR の作成または編集では、必ず次の順序で進めます。

1. ファイル編集機能でドラフトファイルを作成または更新する（`gh` は先に実行しない）
2. ドラフト内容をユーザーに提示し、明示的な承認を得る
3. 承認後にのみコマンドを実行する。作成は `gh pr create`、編集は `gh api` の REST 呼び出しを優先する

承認ステップは絶対に省略しないでください。

## 内容正確性ルール

- PR 本文には、何を・なぜ変更したかを記載する。実装の詳細は diff で確認できるため詳細に記載しない
- 実際のコミット内容に基づいて記述し、含まれていない変更を勝手に追加しない
- 重要事項が不足していると判断した場合は、追加前にユーザーへ確認する

## 手順

1. `git log --oneline origin/main..HEAD` でコミット内容を確認する
2. 変更内容から適切な type を決める：
   | type | 目的 |
   |---|---|
   | `feat` | 新機能 |
   | `fix` | バグ修正 |
   | `docs` | ドキュメントのみの変更 |
   | `style` | コードの意味に影響しない変更 |
   | `refactor` | バグ修正でも新機能追加でもないコード変更 |
   | `perf` | パフォーマンス改善 |
   | `test` | テストの追加または変更 |
   | `chore` | ビルドプロセスや補助ツールの変更 |
   | `ci` | CI 設定の変更 |
3. PR タイトルを決める
   - チケットあり：`[TicketID] <type>: <Summary>`
   - チケットなし：`<type>: <Summary>`
4. `.github/pull_request_template.md` があればテンプレートに従って `personal_workspace/pr_body_<branch-name>.md` を作成する
   <!-- TODO: プロジェクトの記述言語に合わせて編集してください -->
   <!-- 例: 英日バイリンガルの場合、各セクションを <English> --- <日本語> の構成にする など -->
   - PR が issue をクローズする場合、クローズ参照は **本文の最後尾のみ** に配置する。Overview セクションに置いたり、同じ参照を重複記載しないこと
   - **Changes セクション**：実装の「方法」ではなく、何を・なぜ変更したかを書く。実装詳細は diff で確認できるため重複させない
5. 生成した本文ファイルをユーザーに提示し、明示承認を得る
6. `gh pr create` で PR を作成する
   ```bash
   gh pr create --title "<PR title>" --body-file personal_workspace/pr_body_<branch-name>.md
   ```

## 出力ルール

ドラフトファイルを作成または編集したら、ユーザーがすぐ開けるよう、チャット末尾に必ず Markdown リンクでファイルパスを表示します。
