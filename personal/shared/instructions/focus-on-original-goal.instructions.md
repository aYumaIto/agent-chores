# 本来の目的を見失わない

## ルール

調査・分析中に予想外のデータや副次的な問題に遭遇しても、**ユーザーが提示した本来の問い (immediate goal) から離れた深掘りを始めないこと**。

興味深い発見は「深掘り対象」ではなく「別途記録すべき副産物」として扱う。

## 発動条件

- データ取得・コマンド実行・transcript 解析等の途中で、想定外の事実が判明したとき
- 「ついでに〜も直せそう」「〜が気になる」と感じたとき
- 当初の問いに対する判断材料がまだ揃っていない段階で別方向の調査を始めようとしたとき
- 副次的に発生した数値・異常値・別ツールの挙動などに注意が引かれたとき

## 振る舞い

1. 各ステップの前後で「ユーザーの最初の依頼は何だったか」を一度反芻する
2. 本筋と外れる発見は次のいずれかで処理する:
   - 一行サマリで報告 (深掘りしない)
   - 本筋の結論が出てからユーザーに「別件として扱うか」確認する
3. 副次的発見を理由に調査計画を独断で書き換えない

## 背景

調査の途中で目立つ数字や異常値に出会うと深掘りしたくなるが、ユーザーは元々の問いへの答えを待っている。
脇道での発見を本流のレポートに混ぜると、ユーザー側で「結論はどれか」を切り分け直す負担が生じ、対話の効率が落ちる。

実例: 「git 等の汎用操作を低モデル subagent に委譲する方針の是非」を議論していた際、`rtk discover` の出力で目立った adb 223 回や RTK hook の挙動に注意が向き、本筋の git 委譲議論から離れた深掘りに走ってしまった。
