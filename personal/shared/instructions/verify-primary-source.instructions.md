# 一次情報・実体確認のルール

## 原則

判断・実装の前に、**推測や伝聞 (文章・メモ・思い込み) ではなく一次情報を当たること**。

一次情報には 2 種類ある:
- **外部の一次情報**: 公式ドキュメント、リファレンス、スキーマ
- **手元の一次情報**: ファイルの実体、コマンド出力、設定の現在状態

## ルール A: 外部仕様は公式ドキュメント

ツール・ライブラリ・CLI・フレームワークの**仕様・設定・挙動**に関する問題を扱うとき、推測ベースの試行錯誤に入る前に、まず**公式ドキュメントを参照**すること。

### 発動条件

- ツール / CLI / ライブラリの設定・仕様・glob・path・permission など細かい挙動に関する問題
- 「なぜ X がマッチしないのか」「Y の仕様は？」を調べる必要があるとき
- 設定ファイルの構文・セマンティクスに関する不確実性があるとき
- 仮説立てて修正 → 失敗のループが **2 回続いたら、即ドキュメント参照に切り替える**
- **ツール挙動を前提にした action recommendation をユーザーに出す直前**
  - 例: 「〜を開いてください」「〜にログインしてください」「〜を起動すれば共有されます」「〜は MCP から操作できます」
  - 出力前に、その前提が事実かを最低 1 コマンド（`list_*` / `--version` / `--help` / status 取得 / 該当 MCP tool）で実体確認してから出す
  - 「問題が起きてから」ではなく「推奨を出す直前」に発動させること
- **ツール / プロダクトの機能・概念・対応状況に関する断定的言及**
  - 例: 「X には Y 機能が無い」「X は Z をサポートしていない」「X に <概念> は存在しない」「X は <ファイル形式> を読まない」「X と Y は互換が無い」「X は <仕様> に対応している」
  - 「無い」「対応していない」「存在しない」など**ネガティブ断定は特に検証コストが軽視されがち**で、記憶ベースで書きやすい。意識して止める
  - 比較表・対応表・互換性マトリクスを作るときは、行ごとに一次ソース (公式ドキュメント / `--help` / リファレンス) を当てる
  - 「機能 X はおそらく無い」と推測している自覚がある時点で、公式ドキュメント検索 (WebFetch / WebSearch / external-research subagent) を起動する
  - 不確実なら断定せず「ドキュメント未確認だが、おそらく〜」と明示するか、確認してから書く
- **MCP / 外部サーバーの接続範囲・共有スコープに関する判断**
  - 同一プロファイルか / headless か / ユーザーの既存セッションと共有されるか / どのスコープに登録されているか 等
  - 推測せず list 系 / get 系で実体取得してから前提化する

## ルール B: ローカル状態は実体コマンドで確認

メモ・ドキュメント・チャット履歴に書かれた「ファイルの状態」「既存設定の内容」「インストール済みの有無」「過去の経緯」を信用して次の行動を取る前に、**必ず現在の実体を確認すること**。

文章は書かれた時点の状態を表しており、その後ユーザー操作や別セッションで変わっている可能性が常にある。

### 発動条件

| 文章中の主張 | 必須の確認 |
|---|---|
| 「〜は symlink になっている」 | `ls -la` で `l` フラグ確認、または `readlink` |
| 「〜に設定済み」「〜が登録されている」 | 実ファイルを `Read` または `<tool> list` / `<tool> get` で確認 |
| 「〜は既にインストール済み」 | `which`、`<tool> --version`、`pm list` 等 |
| 「〜という構造になっている」 | `jq 'keys'`、`tree`、実ファイルを読む |
| 「ファイル X に Y を書けばよい」 | スキーマ確認 or テスト書き込みで受理されるか先に検証 |
| 「MCP サーバー〜が起動済み / 利用可能」「〜のセッションを共有できる」 | `list_pages` / `list_*` 等の MCP 接続確認系 tool で実体確認。`/mcp` 表示や記憶情報を鵜呑みにしない |

### 矛盾時の優先順位

文章 (memo / コメント / チャット記録) と実体が食い違う場合、**実体を信じる**。文章はメタ情報として参考にするが、判断材料の主は実体。
古い memo はユーザーに確認の上、後で更新する。
